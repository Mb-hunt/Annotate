What should the “Understand the User Prompt” section contain?
It should briefly explain what the user is asking for by listing the key details and requirements in the prompt. This includes things like the task, any limits (location, time, price, format), and what the final answer is expected to look like. It should not include opinions or solutions, only an understanding of the request. (Note: You should not be checking the responses before filling out this section.)

Example:

For the prompt “Find the cheapest iPhone available at the Apple Store,” the section should include what the response should and should not do:

Should do:
Should identify the lowest-priced iPhone available from the Apple Store
Should limit results to the Apple Store only
Should include enough details to clearly identify the iPhone model
	Should not do:
Should not provide multiple options with higher prices
Should not search in other online stores
Should not give information on other phones that are not iPhones



How should such cases be handled when the response is incomplete due to access issues or missing information from the user, and not a model problem?

If the model asks a clarifying question because required information is missing or inaccessible (for example, requesting the user’s location), the response should be rated “Fully Complete” under the Task Completion criterion. Requesting clarification is the correct procedural action when the task cannot be completed without additional user input and does not indicate a model failure.



The model encountered a CAPTCHA or a security check page. How should this be rated?

Even though a CAPTCHA or a security check is an external restriction, the inability to complete the task is still considered unacceptable AI performance for evaluation purposes. As a result, the response should be rated as follows:

Task Completion: Not at all complete, since the task could not be executed.

Factual Accuracy: Cannot be assessed, as no substantive information was produced.

AI Performance: Unacceptable, because the model failed to navigate past the CAPTCHA and complete the task.



When a prompt asks to check an email or calendar, but the email or calendar content is not present, the link does not work, or you do not have access to it, how should the response be evaluated?

Factual Accuracy
Ignore references to the email or calendar when evaluating the response and mark the criterion as “Cannot Confidently Assess” if there are no other objective, verifiable claims.
Exception:
 If the response explicitly includes a Rater Note instructing you to assume the information is correct (for example:
 “Note to rater: an action card for Google Calendar is embedded here. Pretend the card is correctly inserted.”), then the response should be rated as Completely Accurate.
Task Completion and AI Performance
Referring to the email or calendar should not be considered a hallucination and must not negatively affect Task Completion or overall AI performance scores.

📌 Please follow these streamlined rules:
🔐 Assume Data Access:
 If a user attaches a private source (e.g., Google Calendar), the model can see it even if you can't. Treat provided details (addresses, times) as real information, not fabricated.
🚫 Refusals = Failures:
If a model asks “What is the address?” when a calendar is already linked, it has failed the task. Rate it lower than a model that successfully pulls the data. This is similar to the example in the instructions for responding to an email.
⚖️ Correct Rating:
 If you cannot personally verify private info, rate Factual Accuracy as “Cannot confidently assess.”
 Never select “Inaccurate” just because the source is private.
🧠 Check Logic:
 Only flag a hallucination if there is a logical contradiction (e.g., the model claims Feb 6th is a Tuesday, but it’s actually a Friday).
📝 Justification Tip:
“The agent has access to a private source; details provided are assumed correct as there are no logical inconsistencies.”

When we do not know the current date or the location of the user, how should the response be evaluated?

On certain occasions, the agent will make statements that do not necessarily appear to be inaccurate, such as statements related to user elements that may not be accessible for public investigation. In that case, instead of simply assuming that they cannot be verified, the correct question will be, which elements provide certainty and can be verified? e.g., sometimes agents respond with “Wednesday” instead of “Thursday”, or the text states “in the morning” when the appointment is actually “in the afternoon.” 

This can be verified by checking that the time slot, day of the week, and sender's name are consistent. Evidence can include a link to World Time Buddy and/or an online calendar. Remember, in some cases, it will be important to focus on what we can verify, rather than what we lack evidence for, and this is made clear through the justification.

Justification Structure example: 

“The agent has access to a calendar and schedule that is not public; however, there is no evidence of any inconsistencies, such as the day of the week, time, and sender.”

AI Performance:

AI Performance justifications should reference the following:

1. Navigation 🗺️: Was the path logical and direct, or did it meander to irrelevant pages?
2. Interaction 🖱️: Did it click the correct buttons and links precisely, or did it struggle and make mistakes?
3. Efficiency ⏱️: Were there wasted steps, loops, or unnecessary actions?
4. Output Quality 🧹: Is the final scraped text clean and well-formatted, or messy and filled with junk?

This is provided as guidance and is NOT mandatory to follow exactly when writing your justification for AI performance. AI Performance should be lowered for inefficient or illogical navigation, cluttered output, or repeated unnecessary steps.

When a task is blocked by an issue, such as a CAPTCHA page, bot detection, or a broken link that prevents completing the task, then it should be marked as “Yes” for Task blocked by an unusual error or crashes.




What is the Golden Rule? 

Golden Rule #1: CORRECTNESS IS KING
Task Completion and Factual Accuracy are almost always more important than AI Performance. A clumsy response that gets the right answer beats a graceful response with the wrong answer.

Golden Rule #2: PERFORMANCE IS THE TIE-BREAKER
If both responses are correct and complete, use AI Performance to decide which is better. The more efficient and skillful execution wins.

When a prompt is missing critical context such that evaluating the prompt and responses is not possible, the task must be reported and not completed 🚫

How should Factual Accuracy be evaluated for itinerary planning tasks that appear to have been planned in the past?
Any task related to itinerary planning that appears to have been planned in the past, including destinations, transportation (flights, trains, or cars), accommodation details, and schedules for activities or meals, should be rated as “Cannot Confidently Assess” under Factual Accuracy, as the information cannot be independently verified.


How should tasks be evaluated when a response includes coding even though the prompt did not ask for code, or when a response is broken?
If a task has a response that outputs code when it is not asked for, or when one of the responses is broken, this is not a broken task and should not be reported. Such responses should be treated as poor-quality responses.
You should not skip the task. Proceed with rating the task normally as instructed and assign lower scores to responses that produce code or are otherwise broken.



How should tasks be evaluated when the prompt shows “User attached this website to their question: about:blank”?
If a task shows “User attached this website to their question: about:blank”, this is not a broken task and should not be reported. You should treat it as if no attachment was provided by the user. Proceed with rating the task normally as instructed.

Link Evaluation Guidelines
How should embedded links in model responses be handled? 
1. What is a broken link?
A broken link is the same as an invalid link.
A link is considered broken/invalid if:
It does not load at all
It shows 404 / Page Not Found
It shows a server error or any error message that prevents the page from loading

2. If a link opens a Google search page, is it broken?
No, not automatically. Here’s the exact logic:
If clicking an embedded link opens a Google search page, and the actual URL appears in the search bar, then you must copy that URL and paste it directly into the browser.


If that direct link:
Loads successfully → the link is VALID
Shows 404 / error / does not load → the link is BROKEN
So yes, Google search result links can still be valid.
3. Are links that hit CAPTCHA or security pages broken (NOTE: This refers only to the link embedded in the model’s RESPONSE, not to screenshots showing CAPTCHA or security pages encountered by the model.)?
No. They are still considered valid. CAPTCHA or security checks are on the service provider side. If the page loads and works after solving CAPTCHA, the link is valid.
👉 Do NOT mark CAPTCHA/security-page links as broken or invalid

4. Link Evaluation Guidelines 

Type of Link
Definition
Action to Take
✅ Valid Link
The page loads successfully when clicked or when the embedded URL is copied and opened directly in the browser. This includes pages that show CAPTCHA or security checks that can be solved to access the content. 
👍 Do not penalize. Link is considered working and valid.
❌ Broken / Invalid Link
The link does not load at all or shows an error message such as 404 / Page Not Found, server error, or any message indicating the page is unavailable.
⚠️ Penalize under AI Performance, regardless of relevance or user intent.
🔍 Google Search Redirect Link
Clicking the embedded link opens a Google search page, but the actual URL appears in the browser search bar. If that URL loads successfully when opened directly, the link is valid.
📋 Copy the URL and test it directly.

If it loads → ✅ Valid. 
If it errors → ❌ Broken / Invalid.
🎯 Inaccurate Link
The link works but does not point to the correct or specific content required by the prompt (for example, linking to a homepage instead of a requested product page).
🧾 Penalize under Factual Accuracy.
🚫 User -Requested Link (Broken or Inaccurate)
The user explicitly asks for a link, and the provided link is either broken/invalid or inaccurate.
📉 Penalize under Task Completion, since the user intent is not fulfilled.


5. What should you do if the response says:  “Note to rater: a shopping result is embedded here. Pretend the result is correctly inserted here.”
Assume the link is valid and continue evaluating the response as usual.

Handling Refusals
Handling and Evaluating Agent Refusals

Agent refusal scenarios fall into the following categories and should be evaluated accordingly:
1. Invalid refusal

Definition: The agent refuses to perform the task even though it is capable of completing it and provides no valid reason for the refusal.

Prompt: Go to firtnine.ca and put Stream II Helmet into the cart.

 
Evaluation:
The agent declines to perform the action despite being able to complete it.
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)








2. Valid and helpful refusal

Definition: The agent is unable to complete the task due to legitimate reasons such as safety constraints or insufficient information. It clearly explains why it cannot proceed and provides guidance, alternatives, or appropriate clarifying questions.
Prompt: 
Go to myeyebb.com and put Cyber Pink Prescription Cosplay Contacts into the cart.


Evaluation:
The agent explains that the user’s request involves a safety concern and suggests consulting a professional before purchasing the product. While the task is not completed, the agent provides a clear reason and helpful guidance on the next steps.

 • Task Completion: 5 (Fully Complete)
 • Factual Accuracy: 5 (Completely Accurate)
 • AI Performance: 5 (Excellent expert and optimal)






3. Valid but not helpful refusal

Definition: The agent cannot perform the task for legitimate reasons but provides little to no explanation, no useful guidance or next steps, and offers minimal user support.

Prompt: Put a Women's Black & White Skeleton Print Halloween Dress into the cart

Evaluation:
The agent was unable to complete the task because the dress was available in multiple colors and sizes, and the user did not provide those details. However, instead of asking clarifying questions, the agent simply refused the request.
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)


4. Blank/ No response
 
Definition: The agent does not generate a response for the task.
Evaluation:
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)

What should the “Understand the User Prompt” section contain?
It should briefly explain what the user is asking for by listing the key details and requirements in the prompt. This includes things like the task, any limits (location, time, price, format), and what the final answer is expected to look like. It should not include opinions or solutions, only an understanding of the request. (Note: You should not be checking the responses before filling out this section.)

Example:

For the prompt “Find the cheapest iPhone available at the Apple Store,” the section should include what the response should and should not do:

Should do:
Should identify the lowest-priced iPhone available from the Apple Store
Should limit results to the Apple Store only
Should include enough details to clearly identify the iPhone model
	Should not do:
Should not provide multiple options with higher prices
Should not search in other online stores
Should not give information on other phones that are not iPhones



How should such cases be handled when the response is incomplete due to access issues or missing information from the user, and not a model problem?

If the model asks a clarifying question because required information is missing or inaccessible (for example, requesting the user’s location), the response should be rated “Fully Complete” under the Task Completion criterion. Requesting clarification is the correct procedural action when the task cannot be completed without additional user input and does not indicate a model failure.



The model encountered a CAPTCHA or a security check page. How should this be rated?

Even though a CAPTCHA or a security check is an external restriction, the inability to complete the task is still considered unacceptable AI performance for evaluation purposes. As a result, the response should be rated as follows:

Task Completion: Not at all complete, since the task could not be executed.

Factual Accuracy: Cannot be assessed, as no substantive information was produced.

AI Performance: Unacceptable, because the model failed to navigate past the CAPTCHA and complete the task.



When the prompt says “check my email” and the email is not present or the link does not work, how should the response be evaluated?

You may assume these details to be true for Task Completion. For factual accuracy, Ignore these claims when evaluating the model’s response and treat them as subjective. Factual accuracy should be marked as “Cannot confidently assess” if there are no other objective claims in the response.



When we do not know the current date or the location of the user, how should the response be evaluated?

On certain occasions, the agent will make statements that do not necessarily appear to be inaccurate, such as statements related to user elements that may not be accessible for public investigation. In that case, instead of simply assuming that they cannot be verified, the correct question will be, which elements provide certainty and can be verified? e.g., sometimes agents respond with “Wednesday” instead of “Thursday”, or the text states “in the morning” when the appointment is actually “in the afternoon.” 

This can be verified by checking that the time slot, day of the week, and sender's name are consistent. Evidence can include a link to World Time Buddy and/or an online calendar. Remember, in some cases, it will be important to focus on what we can verify, rather than what we lack evidence for, and this is made clear through the justification.

“The agent has access to a calendar and schedule that is not public; however, there is no evidence of any inconsistencies, such as the day of the week, time, and sender.”

Link Evaluation Guidelines
How should embedded links in model responses be handled? 
1. What is a broken link?
A broken link is the same as an invalid link.
A link is considered broken/invalid if:
It does not load at all
It shows 404 / Page Not Found
It shows a server error or any error message that prevents the page from loading

2. If a link opens a Google search page, is it broken?
No, not automatically. Here’s the exact logic:
If clicking an embedded link opens a Google search page, and the actual URL appears in the search bar, then you must copy that URL and paste it directly into the browser.


If that direct link:
Loads successfully → the link is VALID
Shows 404 / error / does not load → the link is BROKEN
So yes, Google search result links can still be valid.
3. Are links that hit CAPTCHA or security pages broken (NOTE: This refers only to the link embedded in the model’s RESPONSE, not to screenshots showing CAPTCHA or security pages encountered by the model.)?
No. They are still considered valid. CAPTCHA or security checks are on the service provider side. If the page loads and works after solving CAPTCHA, the link is valid.
👉 Do NOT mark CAPTCHA/security-page links as broken or invalid

4. Link Evaluation Guidelines 

Type of Link
Definition
Action to Take
✅ Valid Link
The page loads successfully when clicked or when the embedded URL is copied and opened directly in the browser. This includes pages that show CAPTCHA or security checks that can be solved to access the content. 
👍 Do not penalize. Link is considered working and valid.
❌ Broken / Invalid Link
The link does not load at all or shows an error message such as 404 / Page Not Found, server error, or any message indicating the page is unavailable.
⚠️ Penalize under AI Performance, regardless of relevance or user intent.
🔍 Google Search Redirect Link
Clicking the embedded link opens a Google search page, but the actual URL appears in the browser search bar. If that URL loads successfully when opened directly, the link is valid.
📋 Copy the URL and test it directly.

If it loads → ✅ Valid. 
If it errors → ❌ Broken / Invalid.
🎯 Inaccurate Link
The link works but does not point to the correct or specific content required by the prompt (for example, linking to a homepage instead of a requested product page).
🧾 Penalize under Factual Accuracy.
🚫 User -Requested Link (Broken or Inaccurate)
The user explicitly asks for a link, and the provided link is either broken/invalid or inaccurate.
📉 Penalize under Task Completion, since the user intent is not fulfilled.


5. What should you do if the response says:  “Note to rater: a shopping result is embedded here. Pretend the result is correctly inserted here.”
Assume the link is valid and continue evaluating the response as usual.

Handling Refusals
Handling and Evaluating Agent Refusals

Agent refusal scenarios fall into the following categories and should be evaluated accordingly:
1. Invalid refusal

Definition: The agent refuses to perform the task even though it is capable of completing it and provides no valid reason for the refusal.

Prompt: Go to firtnine.ca and put Stream II Helmet into the cart.

 
Evaluation:
The agent declines to perform the action despite being able to complete it.
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)








2. Valid and helpful refusal

Definition: The agent is unable to complete the task due to legitimate reasons such as safety constraints or insufficient information. It clearly explains why it cannot proceed and provides guidance, alternatives, or appropriate clarifying questions.
Prompt: 
Go to myeyebb.com and put Cyber Pink Prescription Cosplay Contacts into the cart.


Evaluation:
The agent explains that the user’s request involves a safety concern and suggests consulting a professional before purchasing the product. While the task is not completed, the agent provides a clear reason and helpful guidance on the next steps.

 • Task Completion: 5 (Fully Complete)
 • Factual Accuracy: 5 (Completely Accurate)
 • AI Performance: 5 (Excellent expert and optimal)






3. Valid but not helpful refusal

Definition: The agent cannot perform the task for legitimate reasons but provides little to no explanation, no useful guidance or next steps, and offers minimal user support.

Prompt: Put a Women's Black & White Skeleton Print Halloween Dress into the cart

Evaluation:
The agent was unable to complete the task because the dress was available in multiple colors and sizes, and the user did not provide those details. However, instead of asking clarifying questions, the agent simply refused the request.
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)


4. Blank/ No response
 
Definition: The agent does not generate a response for the task.
Evaluation:
 • Task Completion: 1 (Not at all Complete)
 • Factual Accuracy: 0 (Cannot Confidently Assess)
 • AI Performance: 1 (Unacceptable, broken, or nonsensical)

Creating Evaluation Guidelines for Browser Agents

Objective
Your task is to define the "Gold Standard" for how the AI agent should respond to a specific user prompt.
A well crafted response should:
Provide sources whenever is reasonable. Sources may come in the form of a product card, link to a website, or an open tab (shown as a screenshot below the response).
When users ask for a list of options, it should include a reasonably sized list. Use your own judgement to decide if the list of options is either too long or too short based on the phrasing of the user prompt.
Adhere to all of the search constraints specified by the user. If someone asks for a "recommendation for an action movie from the 90s featuring Tom Cruise," that includes three constraints for the agent to follow: genre (action), time period (90s), and actor (Tom Cruise).
Stay on topic. It should not include off-topic or extraneous information.
A good guideline establishes the criteria for a good response for a given user prompt. It is Objective, Binary, and Comprehensive.
Objective: Avoid vague phrases like "The response should be helpful." Instead, say "The response must list 3 specific stores."
Binary: A third party should be able to read your guideline and the AI response and answer "Yes" or "No" without debate.
Comprehensive: If the user asked for a specific date, price, location, or format, your guidelines must mention it.
⭐ Use the guides provided in these instructions to help you.
💡 Top Tip! It can be helpful to take notes during your evaluation! You can use our handy note-taking feature.
You will need to refer to some of this work in later parts of the task, taking notes will be very helpful!
Break down the user prompt:
Break the user's prompt down into specific constraints. Look for:
Location: Cities, neighborhoods, specific radiuses, or "near me."
Time: Dates, times of day, "next Friday," "open now."
Quantity: "Top 3," "under $50," "party of 4," "size 6."
Format: Tables, lists, maps, email drafts, calendar events.
Action: Research, booking (or not booking), adding to cart, navigating.

🤖 AI Performance

This question is different from the others. We're not just looking at the final answer—we're judging the skill and intelligence the AI showed along the way. Think of yourself as a performance reviewer watching the AI work.
The key question is: Did the AI achieve the goal gracefully and efficiently, or did it stumble and get lucky?
✅ Key Areas to Observe
When evaluating performance, keep an eye on these four behaviors:
🗺️ Navigation: Was the path logical and direct, or did it meander to irrelevant pages?
🖱️ Interaction: Did it click the correct buttons and links precisely, or did it struggle and make mistakes?
⏱️ Efficiency: Were there wasted steps, loops, or unnecessary actions?
🧹 Output Quality: Is the final scraped text clean and well-formatted, or messy and filled with junk?
⭐ How to Apply the 5-Point Scale
Here's a breakdown of what each rating means, with concrete examples. Click on each rating to see the details.
(5) Excellent (Expert & Optimal)
Ask yourself: Was this a flawless, expert-level performance?
Look for: Direct navigation, precise clicks, no wasted steps, and a clean final output. The AI behaved exactly as a skilled human would.
(4) Good (Effective but with minor flaws)
Ask yourself: Was it almost perfect, but with one small mistake?
Look for: A solid and logical process that had one minor, observable flaw. For example, it took one extra, unnecessary click, or the final output had a tiny formatting error.
(3) Acceptable (Clumsy but Functional)
Ask yourself: Did it get the job done, but in a clumsy or inefficient way? This is the "it works, but..." category.
Look for: An inefficient path (e.g., going to the Help page before finding the Contact page), clumsy actions (e.g., clicking an ad before correcting itself), or a messy but still usable final output.
(2) Poor (Illogical & Flawed)
Ask yourself: Was the process itself a failure, even if it got a tiny bit of the answer right?
Look for: Illogical steps where the AI seems confused or lost. It might struggle repeatedly to find the right button, or it only succeeded by chance, not by skill.
(1) Unacceptable (Broken or Nonsensical)
Ask yourself: Did the AI completely fail to perform the actions required?
Look for: Getting stuck on a page, being caught in a loop, timing out, or being fundamentally incapable of interacting with the website correctly.

🧹 Output Quality: Is the final scraped text clean and well-formatted, or messy and filled with junk?

⭐ How to Apply the 5-Point Scale
Here’s a breakdown of what each rating means, with concrete examples. Click on each rating to see the details.
(5) Excellent (Expert & Optimal)
Ask yourself: Was this a flawless, expert-level performance?
Look for: Direct navigation, precise clicks, no wasted steps, and a clean final output. The AI behaved exactly as a skilled human would.
(4) Good (Effective but with minor flaws)
Ask yourself: Was it almost perfect, but with one small mistake?
Look for: A solid and logical process that had one minor, observable flaw. For example, it took one extra, unnecessary click, or the final output had a tiny formatting error.
(3) Acceptable (Clumsy but Functional)
Ask yourself: Did it get the job done, but in a clumsy or inefficient way? This is the "it works, but..." category.
Look for: An inefficient path (e.g., going to the Help page before finding the Contact page), clumsy actions (e.g., clicking an ad before correcting itself), or a messy but still usable final output.
(2) Poor (Illogical & Flawed)
Ask yourself: Was the process itself a failure, even if it got a tiny bit of the answer right?
Look for: Illogical steps where the AI seems confused or lost. It might struggle repeatedly to find the right button, or it only succeeded by chance, not by skill.
(1) Unacceptable (Broken or Nonsensical)
Ask yourself: Did the AI completely fail to perform the actions required?
Look for: Getting stuck on a page, being caught in a loop, timing out, or being fundamentally incapable of interacting with the website correctly.
Important: If the response gets stuck on a CAPTCHA or security check it is always rated as Unacceptable (1/5) performance. See the FAQ: CAPTCHA Handling for a visual example and more details.

 Justification with limited/restricted information 

On certain occasions, the agent will make statements that do not necessarily appear to be inaccurate, such as statements related to user elements that may not be accessible for public investigation. In that case, instead of simply assuming that they cannot be verified, the correct question will be, which elements provide certainty and can be verified? e.g., sometimes agents respond with “Wednesday” instead of “Thursday”, or the text states “in the morning” when the appointment is actually “in the afternoon.” 

This can be verified by checking that the time slot, day of the week, and sender's name are consistent. Evidence can include a link to World Time Buddy and/or an online calendar. Remember, in some cases, it will be important to focus on what we can verify, rather than what we lack evidence for, and this is made clear through the justification.

Justification Structure example: 

“The agent has access to a calendar and schedule that is not public; however, there is no evidence of any inconsistencies, such as the day of the week, time, and sender.”

Time-dependent Responses:
	The timestamp of the resources and responses mentioned is the key drive here; do you have some insight into what timeline for reference is used? On the other hand, each batch could have a certain timeline reference for the client, and this must be clarified.
	
Stay focused: 
Be sure not to assume that the structure of the response actually refers to a matter related to a specific period. For example, sometimes references are passing and are not really a reference to be investigated. 

“Get the price of the blue vase we bought on Saturday” is not the same as “Search for more blue vases at a lower price than on Saturday.”

In the first case, it is sufficient to obtain a reference price for the same item on the current date. In the second case, a comparison must be made between a previous price and the current price. In either case, it is sufficient to obtain a price reference; a specific amount is not necessary, but rather whether the item costs less.

Flight/Hotel history research tools: 
Can you use tools like this, or similar ones, to track the history of certain flight and hotel prices? In such cases, a clear reference to the source is key to staying objective; it isn´t an assumption.

Section 1: Understand the User Prompt (0-5) ⭐ NEW
What This Measures: Whether the trainer created a comprehensive constraint checklist from the user prompt BEFORE evaluating responses.

What Trainers Must Do
Before looking at Response A or B, trainers must create a checklist answering:

This creates a checklist describing what the agent SHOULD DO and SHOULD NOT DO.

How should embedded links in model responses be handled? 
Note: If the user did not ask for a link, the agent does NOT have to provide a link in the response.
1. What is a broken link? A broken link is the same as an invalid link. A link is considered broken/invalid if:
It does not load at all
It shows 404 / Page Not Found
It shows a server error or any error message that prevents the page from loading
2. If a link opens a Google search page, is it broken?
No, not automatically. Here’s the exact logic: If clicking an embedded link opens a Google search page, and the actual URL appears in the search bar, then you must copy that URL and paste it directly into the browser. Example link.

[Slide 11]

If that direct link:
Loads successfully → the link is VALID
Shows 404 / error / does not load → the link is BROKEN
So yes, Google search result links can still be valid.
3. Are links that hit CAPTCHA or security pages broken (NOTE: This refers only to the link embedded in the model’s RESPONSE, not to screenshots showing CAPTCHA or security pages encountered by the model.)?
No. They are still considered valid. CAPTCHA or security checks are on the service provider side. If the page loads and works after solving CAPTCHA, the link is valid.
👉 Do NOT mark CAPTCHA/security-page links as broken or invalid
4. 5. What should you do if the response says:  “Note to rater: a shopping result is embedded here. Pretend the result is correctly inserted here.”
Assume the link is valid and continue evaluating the response as usual. This link should be marked as factually accurate.

Edge Cases: Cannot Confidently Assess
1. Subjective-Only Responses
Why : The response contains no objective, verifiable claims.Example
“This tutorial is very easy to follow.”
2. Private, Missing, or Inaccessible Information
Why: Evidence may exist but cannot be accessed by the evaluator.Examples
“According to my private Google Calendar, the meeting is at 3 PM.”
“The email confirms the update.” (email not provided)
3. Broken Links or Blocked Content
Why: Verification is prevented by technical restrictions.Examples
Links that return 404 errors
Pages blocked by CAPTCHA or login walls

4. Past Itinerary Planning or Personal History
Why: Past travel or personal events cannot be independently verified.Examples
“I stayed near Times Square during my trip last year. What was the price of the hotel?”
“We took a train from Rome to Florence last summer. For May 23 and 24, what hotels were available in Florence?”
5. Agent Action Claims Without Verifiable Evidence
Why: The response claims an action occurred, but no proof is available.Examples

“The form was successfully submitted.” (no visible evidence)
6. Image Prompts
Why: The links in the response do not workExamples
“Here is the image you requested.” (The provided link does not work)

Edge Cases: Cannot Confidently Assess (Refusals)
Agent refusal scenarios fall into the following categories and should be evaluated accordingly:
1. Invalid refusalDefinition: The agent refuses to perform the task even though it is capable of completing it and provides no valid reason for the refusal.
Prompt: Go to firtnine.ca and put Stream II Helmet into the cart.
Respone B: Let's talk about something else. That's not the type of thing I can help you with.
Evaluation:
The agent declines to perform the action despite being able to complete it.
 • Task Completion: 1 (Not at all Complete) • Factual Accuracy: 0 (Cannot Confidently Assess) • AI Performance: 1 (Unacceptable, broken, or nonsensical)

[Slide 18]

Edge Cases: Cannot Confidently Assess (Refusals)
2. Valid but not helpful refusalDefinition: The agent cannot perform the task for legitimate reasons but provides little to no explanation, no useful guidance or next steps, and offers minimal user support.
Prompt: Put a Women's Black & White Skeleton Print Halloween Dress into the cart
Response A: I am sorry, I am not able to complete the task.
Evaluation:
The agent was unable to complete the task because the dress was available in multiple colors and sizes, and the user did not provide those details. However, instead of asking clarifying questions, the agent simply refused the request.
 • Task Completion: 1 (Not at all Complete) • Factual Accuracy: 0 (Cannot Confidently Assess) • AI Performance: 1 (Unacceptable, broken, or nonsensical)
3. Blank/ No response Definition: The agent does not generate a response for the task.
Evaluation: • Task Completion: 1 (Not at all Complete) • Factual Accuracy: 0 (Cannot Confidently Assess) • AI Performance: 1 (Unacceptable, broken, or nonsensical)


[Slide 19]

Edge Cases: Cannot Confidently Assess (Refusals)
4. Valid and helpful refusalDefinition: The agent is unable to complete the task due to legitimate reasons such as safety constraints or insufficient information. It clearly explains why it cannot proceed and provides guidance, alternatives, or appropriate clarifying questions.
Prompt: Go to myeyebb.com and put Cyber Pink Prescription Cosplay Contacts into the cart.





Evaluation:
The agent explains that the user’s request involves a safety concern and suggests consulting a professional before purchasing the product. While the task is not completed, the agent provides a clear reason and helpful guidance on the next steps.
 • Task Completion: 5 (Fully Complete) • Factual Accuracy: 5 (Completely Accurate) • AI Performance: 5 (Excellent expert and optimal)

What Happens to Task Completion When FA is Cannot Confidently Assess?
Do not assign Fully Complete (5)

🔑 Core Rule:
If Factual Accuracy is rated as Cannot Confidently Assess for a critical claim, Task Completion cannot be Fully Complete.


[Slide 21]

Factual Accuracy and Task Completion Alignment for Evidence-Based Tasks
✅ Case 1: Valid Evidence Provided (Screenshot or Link)
Example: “I have added the Palm Golf Co. Icon Snapback Hat to your cart on palmgolfco.com.”
If a valid screenshot or link clearly verifies the action:
Factual Accuracy: Fully Accurate
Task Completion: Fully Complete
⚖ Case 2: Evidence Partially Verifies the Claim
If the screenshot or link does not fully confirm completion, penalize proportionally based on the weight of the claim.
Example: Prompt: Find the product and add it to the cart Response: Shows a screenshot of the product, but it is not added to the cart
Factual Accuracy: Cannot Confidently Assess
Task Completion: Partially Complete


[Slide 22]

Factual Accuracy and Task Completion Alignment for Evidence-Based Tasks
❌ Case 3: No Evidence Provided (Only AI Claim)
If the response claims completion but provides no screenshot, link, or verifiable proof:
Factual Accuracy: Cannot Confidently Assess (for the unverifiable claim)
Task Completion: 3–4 (Partially or Mostly Complete)
🚫 Case 4: Screenshot Is Irrelevant to the Task
If the provided screenshot does not relate to the claimed action:
Factual Accuracy: Cannot Confidently Assess
Task Completion: Not at All Complete
Overall performance reflects poor verification and execution.
📌 Applies To:
Add-to-cart actions
Email sent claims
Calendar event creation
Dynamic prices or availability
Any external action that cannot be verified

Assessing Overall Factual Accuracy
When evaluating overall accuracy across multiple claims:
If there is an equal split (for example, 4 Accurate claims and 4 Cannot Confidently Assess claims), the rating should reflect Mixed Accuracy.
General Rule for Mixed Claims:
If most claims are Accurate, rate as Mostly Accurate.
If most claims are Inaccurate, rate as Mostly Inaccurate.
If there is an equal balance between Accurate and Cannot Confidently Assess, rate as Mixed Accuracy.
If the majority of major or critical claims are Cannot Confidently Assess, lean toward the level closest to Cannot Confidently Assess based on claim weight.
⚖ Important: Always consider the weight and importance of the claims, not just the count. A single critical claim can outweigh multiple minor ones.
